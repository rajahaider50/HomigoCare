SECTION 5: REALTIME DATABASE - COMPLETE SETUP
================================================================================

Step 1: Firebase Console → Build → Realtime Database → Create database

Step 2: Choose location: asia-south1 (Mumbai)
   - This is closest to Pakistan
   - Lowest latency

Step 3: Security Rules (IMPORTANT!)

   You have TWO modes:

   =============================================
   MODE 1: TESTING MODE (Development)
   =============================================
   
   Use this while developing the app. Allows all reads/writes.
   
   {
     "rules": {
       ".read": true,
       ".write": true
     }
   }
   
   WARNING: This allows ANYONE to read/write your database.
   Only use during development. NEVER in production.
   
   How to enable:
   - When you first create the database, Firebase asks:
     "Start in locked mode or test mode?"
   - Choose "Test mode"
   - Rules will be: { "rules": { ".read": true, ".write": true } }
   - Click "Enable"

   =============================================
   MODE 2: LOCKED MODE (Production)
   =============================================
   
   This is the secure mode with proper rules.
   
   Use the rules from: real-time-database-security-rule.txt
   
   How to switch:
   - Go to Realtime Database → Rules tab
   - Paste the production rules
   - Click "Publish"
   
   WARNING: Make sure your code is ready before switching.
   If rules are too strict, app will stop working.

Step 4: Database URL:
   After creation, you'll get a URL like:
   https://homigo-care-XXXXX-default-rtdb.firebaseio.com/
   
   Note this URL down - it's used in the app code.

SECTION 7: TESTING MODE vs LOCKED MODE
================================================================================

   =============================================
   TESTING MODE (Development Phase)
   =============================================
   
   When to use: While building and testing the app
   Rules: { "rules": { ".read": true, ".write": true } }
   
   Pros:
   - Anyone can read/write (easy testing)
   - No authentication required for DB access
   - Fast development
   
   Cons:
   - SECURITY RISK - anyone can delete your data
   - Anyone can inject malicious data
   - NEVER use in production
   
   How to set:
   1. Firebase Console → Realtime Database → Rules
   2. Paste: { "rules": { ".read": true, ".write": true } }
   3. Click "Publish"
   
   =============================================
   LOCKED MODE (Production)
   =============================================
   
   When to use: After app is ready and tested
   Rules: Use the rules from real-time-database-security-rule.txt
   
   Pros:
   - Only authenticated users can access
   - Role-based access control
   - Users can only modify their own data
   - Admin has full access
   
   Cons:
   - Must be properly configured
   - Wrong rules = app breaks
   
   How to switch:
   1. Make sure all auth is working
   2. Make sure your code handles errors
   3. Go to Realtime Database → Rules
   4. Paste production rules
   5. Click "Publish"
   6. Test the app thoroughly

SECTION 11: COMMON ERRORS AND FIXES
================================================================================

Error: "Firebase has not been initialized"
Fix: Make sure Firebase.initializeApp() is called in main.dart before runApp()

Error: "Database not found"
Fix: Check databaseURL in firebase_options.dart matches your Firebase Console URL

Error: "Permission denied"
Fix: Check security rules. For testing, use test mode rules.

Error: "google-services.json not found"
Fix: Make sure file is at android/app/google-services.json (not in lib/)

Error: "SHA-1 mismatch"
Fix: Run: keytool -list -v -keystore ~/.android/debug.keystore -alias androiddebugkey -storepass android
     Add SHA-1 to Firebase Console → Project Settings → Your App

Error: "Network error"
Fix: Check internet permission in AndroidManifest.xml
