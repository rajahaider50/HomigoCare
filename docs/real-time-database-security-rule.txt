SECTION 2: RULES EXPLAINED - WHO CAN DO WHAT
================================================================================

USER ROLES:
- patient: Can book appointments, chat, view own data
- doctor: Can manage appointments, view own profile, chat with patients
- nurse: Can manage home visits, view own profile
- lab: Can manage test orders, view own profile
- admin: Full access to everything

PERMISSION MATRIX:
================================================================================
| Path               | Patient | Doctor | Nurse | Lab   | Admin       |
|--------------------|---------|--------|-------|-------|-------------|
| /users/self        | R/W     | R/W    | R/W   | R/W   | R/W         |
| /users/other       | None    | None   | None  | None  | R/W         |

SECTION 3: DETAILED RULES BREAKDOWN
================================================================================

1. USERS PATH:
   
   Users can READ their own data only.
   Users can UPDATE their own profile (name, phone, profileImage).
   Admin can READ and WRITE all user data.
   Admin can change user status (active/inactive).
   Admin can change user role.

2. DOCTORS PATH:
   
   Any authenticated user can READ doctor profiles (for listing).
   Doctor can UPDATE their own profile.
   Only Admin can:
     - Change status (pending → approved → rejected)
     - Toggle isVerified
     - View all doctor documents
   Doctor documents are private - only doctor and admin can access.

3. NURSES PATH:
   
   Same as doctors.
   Any authenticated user can READ nurse profiles.
   Nurse can UPDATE own profile.
   Admin has full control.

4. LABS PATH:
   
   Same as doctors/nurses.
   Any authenticated user can READ lab profiles.
   Lab can UPDATE own profile.
   Admin has full control.

5. BOOKINGS PATH:
   
   Patient can CREATE bookings for themselves.
   Patient can READ own bookings.
   Provider (doctor/nurse/lab) can READ bookings assigned to them.
   Provider can UPDATE booking status (confirm/complete).
   Patient can CANCEL their own booking.
   Admin can READ and WRITE all bookings.

6. CHATS PATH:
   
   Patient and Provider can READ chat messages between them.
   Either party can SEND messages.
   Only the sender can update their own message (mark as read).
   Admin can READ all chats (for monitoring).

7. MCQs PATH:
   
   Only Admin can READ and WRITE questions.
   Doctor can READ results (only their own).
   Doctor can WRITE their own test result.
   Admin can READ and WRITE all results.

8. SETTINGS PATH:
   
   Everyone can READ settings.
   Only Admin can WRITE/UPDATE settings.

SECTION 6: RULE VALIDATION SYNTAX
================================================================================

Firebase Realtime Database rules use JavaScript-like syntax:

.auth != null          → User must be logged in
auth.uid               → Current user's unique ID
data.child('x').val()  → Value of field 'x' in current node
newData.child('x').val() → Value of field 'x' being written
root.child('users').child(auth.uid).child('role').val()
                       → Get current user's role from users table
newData.isString()     → Check if value is a string
newData.isNumber()     → Check if value is a number
newData.isBoolean()    → Check if value is a boolean
newData.isList()       → Check if value is an array
newData.val().length   → Get string length
.indexOf() !== -1      → Check if value exists in array
